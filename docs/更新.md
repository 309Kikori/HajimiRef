# Hajimi Ref 更新说明

## 2026-03-11 周二 - Win 端双击组名重命名修复

### 下午 - 修复：双击组名标签无法触发重命名（缩放后失效）
- **变更动机**：
  - Win 版本中双击组名标签应弹出重命名对话框，但在画布缩放后双击完全无响应
  - 该功能此前已实现，但由于碰撞检测区域与实际绘制位置不匹配，导致缩放后失效
  - 用户在正常使用中几乎必然会缩放画布，因此该 Bug 影响面极广
- **问题根因**：`_name_label_local_rect()` 与 `paint()` 的标签区域在缩放后严重不匹配
  | 维度 | `_name_label_local_rect()`（碰撞检测） | `paint()`（实际绘制） |
  |------|------|------|
  | 标签高度 | `font_size + 20`（固定值） | `fm.height() + padding_v * 2`（除以 `lod`） |
  | 标签宽度 | `max(rect.width(), 200)`（固定值） | `text_width + padding_h * 2`（除以 `lod`） |
  | 字体大小 | 不涉及 | `font_size / lod` |
  - 当画布缩放时（例如缩小到 50%，`lod=0.5`），实际绘制的标签变成 2 倍大小并向上偏移更多，但碰撞检测区域仍为固定的小区域，**完全对不上**
  - 此外 `GroupItem` 的 `zValue` 为 `-100`（在图片下方），`RubberBandDrag` 模式下双击事件无法直接到达 GroupItem，必须依赖 View 层的主动碰撞检测
- **修复方案**：让碰撞检测感知缩放级别（lod）
  1. **`_name_label_local_rect(lod=None)` 方法重写**：
     - 新增 `lod` 参数，传入时使用与 `paint()` 完全一致的计算逻辑（`QFontMetricsF` + `font_size/lod` + `padding/lod`），精确匹配实际标签位置
     - 不传 `lod` 时（供 `boundingRect` / `shape` 使用），返回足够大的保守区域确保任意缩放下覆盖
  2. **View 层 `mouseDoubleClickEvent` 改进**：
     - 通过 `self.transform().m11()` 获取当前缩放级别
     - 调用 `item._name_label_local_rect(lod)` 进行精确碰撞检测
  3. **GroupItem 自身 `mouseDoubleClickEvent` 同步修复**：
     - 从 `self.scene().views()[0]` 获取当前 lod
     - 同样传入 `_name_label_local_rect(lod)` 精确命中
  4. **新增 `QFontMetricsF` import**：精确模式下需要用 `QFontMetricsF` 计算文字尺寸
- **用户体验效果**：
  - **修复前**：画布缩放后双击组名标签无任何反应，仅在 100% 缩放时偶尔能命中
  - **修复后**：无论画布处于任何缩放级别，双击组名标签都能准确触发重命名对话框
- **影响文件**：
  - Windows: `Views/Canvas.py`（`_name_label_local_rect` 方法重写；View 层 `mouseDoubleClickEvent` 传入 lod；GroupItem 层 `mouseDoubleClickEvent` 获取 lod 并传入；新增 `QFontMetricsF` import）

## 2026-03-11 周二 - macOS Liquid Glass 效果增强 + 撤销系统补全

### 下午 - 优化：增强 Liquid Glass（液态玻璃）效果感知
- **变更动机**：
  - macOS 26 引入的 Liquid Glass 是核心视觉特性，但当前应用中玻璃效果不明显、感知不强
  - 所有 `.glassEffect()` 都是孤立使用的，缺少 `GlassEffectContainer` 包裹，无法形成液态玻璃的核心「联动/融合」效果
  - 画布区域使用深色不透明背景，`glassEffect` 在此背景上视觉感知弱
  - 组名标签 tint 透明度过低（0.3），效果不明显
- **优化内容**：
  1. **HelpView.swift**：用 `GlassEffectContainer` 包裹所有玻璃卡片元素，启用液态玻璃的联动/融合动效，使三张卡片（基本操作、图片操作、快捷键）之间产生视觉关联
  2. **AboutView.swift**：用 `GlassEffectContainer` 包裹整个关于页面；技术栈徽章 tint 从 `.clear` 增强为 `.orange.opacity(0.2)` 使效果更醒目
  3. **SettingsView.swift**：`OtherSettingsView` 占位页面用 `GlassEffectContainer` 包裹；图标增加独立的 `.glassEffect(.regular.tint(.purple.opacity(0.15)), in: .circle)` 装饰；外层容器 tint 从 `.clear` 改为 `.purple.opacity(0.08)`
  4. **HajimiRef_macosApp.swift**：工具栏新增「复制到剪贴板」按钮，增加更多工具栏项以增强 Liquid Glass 材质的可见面积；添加工具栏分组注释
  5. **CanvasView.swift**：组名标签 glassEffect tint 透明度从 `0.3` 提高到 `0.5`，使颜色在深色画布上更醒目
- **GlassEffectContainer 工作原理**：
  - macOS 26 中，`GlassEffectContainer` 将内部所有 `.glassEffect()` 视图纳入同一个玻璃容器
  - 多个玻璃元素在接近时会产生「液态融合」动画，形成类似水银的视觉联动
  - 不使用 `GlassEffectContainer` 时，每个 `.glassEffect()` 都是独立的，无法触发融合效果
- **用户体验效果**：
  - **修改前**：玻璃效果孤立且不明显，tint 颜色在深色画布上几乎不可见
  - **修改后**：多个玻璃元素之间产生联动融合动画，tint 颜色更饱和醒目，工具栏和页面元素的玻璃质感大幅提升
- **影响文件**：
  - macOS: `Views/HelpView.swift`（添加 `GlassEffectContainer` 包裹）
  - macOS: `Views/AboutView.swift`（添加 `GlassEffectContainer` 包裹 + 增强 tint）
  - macOS: `Views/SettingsView.swift`（`OtherSettingsView` 添加 `GlassEffectContainer` + 图标玻璃装饰 + 增强 tint）

### 下午 - 修复：Cmd+Z 撤销系统补全（5 类操作缺失 undo）
- **变更动机**：
  - 用户反馈 Cmd+Z 不能全局撤回，某些操作完全无法撤回
  - 经排查，以下 5 类操作直接修改数据但**完全没有记录 undo**
- **缺失 undo 的操作清单**：
  | # | 操作 | 问题描述 |
  |---|------|---------|
  | 1 | **智能排序** `smartSortSelected()` | 移动了多张图片位置但没有记录 undo |
  | 2 | **去背景** `removeBackground()` | 修改了图片数据但没有记录 undo |
  | 3 | **重置缩放** "Reset Scale" | 直接改 `scale = 1.0` 没有记录 undo |
  | 4 | **组名称编辑** sheet 内修改 | 直接修改 `groups[index].name` 没有记录 undo |
  | 5 | **组调整大小** resizeGesture | 拖拽改组大小没有记录 undo |
- **修复方案**：
  1. **UndoManager.swift 新增 4 个 UndoableAction case**：
     - `replaceImageData(imageId:oldData:newData:)` — 替换图片数据（去背景等）

     - `resizeGroup(groupId:oldRect:newRect:)` — 调整组大小
  2. **AppState.swift — `smartSortSelected()`**：
     - 在 `withAnimation` 块之前收集所有图片的 `oldPosition` → `newPosition`
     - 记录为 `.batchMove(changes:)` 操作（复用已有的批量移动类型）
  3. **AppState.swift — `removeBackground()`**：
     - 在函数开始保存 `oldData = images[index].data`
     - 在 `DispatchQueue.main.async` 数据替换完成后记录 `.replaceImageData`
  4. **CanvasView.swift — "Reset Scale" 按钮**：
     - 保存 `oldScale = images[index].scale` 后再设为 1.0
     - 记录 `.scale(imageId:oldScale:newScale:)` 操作
  5. **CanvasView.swift — 组名称编辑 OK 按钮**：
     - 保存 `oldName` 后再修改
     - 当 `oldName != editingName` 时记录 `.renameGroup`
  6. **CanvasView.swift — 组调整大小 `resizeGesture.onEnded`**：
     - 对比 `resizeStartRect` 和当前组的新尺寸
     - 当两者不同时记录 `.resizeGroup`
- **撤销覆盖率变化**：
  | 阶段 | 覆盖率 | 已支持的操作 |
  |------|--------|-------------|
  | 修复前 | ~70% | 移动、缩放、旋转、删除、添加、图层调整、清空画板、打组/解组 |
  | 修复后 | ~100% | 上述全部 + 智能排序、去背景、重置缩放、组重命名、组调整大小 |
- **用户体验效果**：
  - **修复前**：执行智能排序、去背景等操作后按 Cmd+Z 无响应，用户被迫手动恢复
  - **修复后**：所有用户可感知的操作均可通过 Cmd+Z 撤回，Cmd+Shift+Z 重做
- **影响文件**：
  - macOS: `Models/UndoManager.swift`（新增 `replaceImageData`、`resetScale`、`renameGroup`、`resizeGroup` 四个 action case + 对应的 undo/redo 处理逻辑）
  - macOS: `Models/AppState.swift`（`smartSortSelected` 添加 `batchMove` undo 记录；`removeBackground` 添加 `replaceImageData` undo 记录）
  - macOS: `Views/CanvasView.swift`（"Reset Scale" 按钮添加 `scale` undo 记录；组名编辑 OK 按钮添加 `renameGroup` undo 记录；组调整大小 `onEnded` 添加 `resizeGroup` undo 记录）

## 2026-03-06 周四 - 缩放参考线吸附（双端）

### 下午 17:50 - 功能：缩放图片时也显示智能参考线吸附
- **变更动机**：
  - 此前智能参考线仅在**拖拽移动**图片时生效，缩放（resize）图片时完全没有参考线
  - 用户无法在缩放过程中精确对齐其他图片的边缘，影响排版效率
  - 专业设计软件（如 Figma、Sketch）在缩放时同样提供吸附功能
- **缩放吸附与拖拽吸附的本质区别**：
  | 操作 | 吸附修正对象 | 说明 |
  |------|-------------|------|
  | 拖拽移动 | 位置偏移量（dx, dy） | 整体平移到吸附位置 |
  | 缩放 | 缩放比例（ratio / k） | 反推精确的缩放倍率，使边缘恰好对齐参考线 |
- **核心流程**：
  1. **缩放开始**：调用 `buildSnapGuides` 缓存所有非选中图片的参考线
  2. **缩放过程中**：根据当前 ratio 计算缩放后的边界 → 检测边缘是否接近参考线 → 如果接近，反算精确的 ratio 使边缘对齐 → 更新活跃辅助线用于绘制
  3. **缩放结束**：清除参考线缓存和活跃辅助线
- **反推 ratio 的数学原理**：
  - 设锚点为 `anchor`，原始边缘到锚点的距离为 `orig_dist`，目标参考线到锚点的距离为 `snap_dist`
  - 则修正后的 ratio = `snap_dist / orig_dist`
  - 对 X、Y 两个方向分别计算，取偏差更小的方向优先吸附
- **Win11 端技术实现**：
  - **新增方法**：`RefView._performResizeSnap(anchor, ratio, selected)` — 计算缩放后联合边界，检测 6 条关键线（left/right/centerX/top/bottom/centerY）吸附，反推精确 ratio
  - **修改区域**：
    - `RefItem.mousePressEvent` 缩放分支：缩放开始时调用 `view._buildSnapGuides(dragged_ids)` 缓存参考线
    - `RefItem.mouseMoveEvent` 缩放分支：新增 `_performResizeSnap()` 调用，根据吸附结果修正 ratio
    - `RefItem.mouseReleaseEvent` 缩放分支：清除 `_active_snap_lines`、`_snap_x_guides`、`_snap_y_guides`
- **macOS 端技术实现**：
  - **新增方法**：`AppState.performResizeSnap(anchor:ratio:selectedIds:canvasScale:)` — 计算缩放后联合边界，检测 6 条关键线吸附，反推精确 k 值，生成辅助线
  - **修改区域**：
    - `resizeGesture.onChanged` 初始化块：缩放开始时调用 `buildSnapGuides(draggedIds:)` 缓存参考线
    - `resizeGesture.onChanged` 计算 k 之后：调用 `performResizeSnap()` 修正 k 值并更新活跃辅助线
    - `resizeGesture.onEnded`：清除 `snapXGuides`、`snapYGuides`、`activeSnapLines`
- **用户体验效果**：
  - **修改前**：缩放图片时没有任何参考线，无法精确对齐其他图片边缘
  - **修改后**：缩放过程中图片边缘（left/right/top/bottom/center）自动吸附到其他图片的对应边缘，并显示蓝色虚线辅助线，与拖拽时体验一致
- **影响文件**：
  - Windows: `Views/Canvas.py`（`RefItem.mousePressEvent`、`mouseMoveEvent`、`mouseReleaseEvent` 缩放分支；新增 `RefView._performResizeSnap` 方法）
  - macOS: `Models/AppState+SnapGuides.swift`（新增 `performResizeSnap` 方法）
  - macOS: `Views/CanvasView.swift`（`resizeGesture.onChanged`、`resizeGesture.onEnded` 集成缩放参考线）

## 2026-03-06 周四 - 画布扩展动画（双端）

### 下午 16:30 - 功能：画板边界扩展时平滑动画过渡
- **变更动机**：
  - 双端的画板在图片拖出边界时会自动扩展（只扩展不收缩），但扩展过程是**瞬间跳变**的
  - 边界突然变大会让用户产生视觉上的不适感，影响使用体验
  - 专业工具通常会使用平滑过渡来表达界面区域的变化
- **设计思路**：「显示边界」与「逻辑边界」分离
  - **逻辑边界**（`_board_bounds` / `boardBounds`）：瞬间更新到目标值，用于逻辑计算
  - **显示边界**（`_display_board_bounds` / `displayBoardBounds`）：通过插值动画平滑过渡到逻辑边界，用于实际绘制
  ```
  图片移出画板 → 逻辑边界瞬间扩展 → 启动动画定时器 → 显示边界每帧插值逼近逻辑边界 → 到达目标后停止
  ```
- **Win11 端技术实现**：
  - **新增属性**：`_display_board_bounds`（显示边界）、`_board_anim_timer`（16ms 动画定时器，~60fps）
  - **新增方法**：
    - `_startBoardBoundsAnimation()`：当逻辑边界变化时启动定时器
    - `_animateBoardBounds()`：每帧用 lerp 对四条边分别插值（因子 0.15），收敛阈值 0.5px
  - **修改方法**：
    - `drawBackground()`：使用 `_display_board_bounds` 代替 `_board_bounds` 进行绘制
    - `resetBoardBounds()` / `resetBoardToFitImages()`：重置时同步启动动画过渡
- **macOS 端技术实现**：
  - **新增属性**：`displayBoardBounds`（显示边界）、`boardBoundsAnimationTimer`（Timer，~60fps）
  - **新增方法**：
    - `animateDisplayBoardBounds(to:)`：记录动画起始状态，启动定时器
    - `tickBoardBoundsAnimation(_:)`：每帧用 easeOut cubic 缓动插值 `(1 - (1-t)^3)`，总时长 0.25s
    - `stopBoardBoundsAnimation()`：停止定时器并清理状态
  - **修改方法**：
    - `expandBoardBounds(toInclude:)`：扩展后启动动画
    - `resetBoardBounds()`：重置时启动动画
    - `clearBoard()`：清空画板时直接跳到目标值（无需动画）
  - **修改绘制层**：
    - `ActiveAreaBackground`：`appState.boardBounds` → `appState.displayBoardBounds`
    - `OptimizedGridBackground`：`appState.boardBounds` → `appState.displayBoardBounds`
- **动画参数对比**：
  | 参数 | Win11 | macOS |
  |------|-------|-------|
  | 帧率 | ~60fps（16ms QTimer） | ~60fps（1/60s Timer） |
  | 缓动方式 | 线性 lerp（因子 0.15） | easeOut cubic `1-(1-t)^3` |
  | 完成判定 | 四条边偏差均 < 0.5px | 经过时间 ≥ 0.25s |
  | 重置行为 | 动画过渡 | 动画过渡 |
  | 清空行为 | 直接跳转 | 直接跳转 |
- **用户体验效果**：
  - **修改前**：拖拽图片超出画板时，背景区域瞬间跳变扩大，视觉突兀
  - **修改后**：背景区域平滑地"生长"到新大小，带有自然的缓动效果
- **影响文件**：
  - Windows: `Views/Canvas.py`（新增 `_display_board_bounds`、`_board_anim_timer`、`_startBoardBoundsAnimation`、`_animateBoardBounds`；修改 `drawBackground`、`resetBoardBounds`、`resetBoardToFitImages`）
  - macOS: `Models/AppState.swift`（新增 `displayBoardBounds`、`boardBoundsAnimationTimer` 及动画方法；修改 `expandBoardBounds`、`resetBoardBounds`、`clearBoard`）
  - macOS: `Views/CanvasView.swift`（`ActiveAreaBackground`、`OptimizedGridBackground` 改用 `displayBoardBounds`）

## 2026-03-06 周四 - Windows 端性能优化（5 项修复）

### 下午 15:00 - 优化：全面修复 Win11 版本画布卡顿问题
- **变更动机**：
  - Win11 版本在使用过程中存在莫名卡顿，尤其是拖拽图片、缩放画布时明显
  - 经分析发现 5 个性能瓶颈，分为 P0（严重）、P1（中等）、P2（轻微）三个等级
  - macOS 端有 LOD 点阵自适应特性，Win11 端缺少此优化

#### 🔴 P0 修复1：`FullViewportUpdate` → `NoViewportUpdate` + 手动定时器控制
- **问题描述**：
  - `QGraphicsView.FullViewportUpdate` 模式导致**任何微小变化都触发整个视口的完整重绘**
  - 鼠标移动、吸附辅助线更新、滚动/缩放时都会触发全屏重绘
  - 在 OpenGL 视口上每帧执行 `drawBackground()` → `_updateBoardBounds()` → `_drawGridInRect()` 形成级联重绘链
- **修复方案**：
  - 将 `setViewportUpdateMode` 改为 `NoViewportUpdate`，完全关闭自动重绘
  - 新增 `_viewport_update_timer`（16ms 单次触发定时器，~60fps）
  - 新增 `scheduleViewportUpdate()` 方法，去抖调度视口刷新（16ms 内多次调用只触发一次）
  - 新增 `_doViewportUpdate()` 方法，由定时器回调执行 `viewport().update()`
  - 在所有需要刷新的操作中手动调用 `scheduleViewportUpdate()`：
    - 滚轮缩放、平移拖拽、鼠标移动、鼠标释放、图片添加/删除
- **性能提升**：从「每次事件都全屏重绘」降为「每 16ms 最多重绘一次」

#### 🔴 P0 修复2：`_getGuideExtentMin/Max` 全量遍历 → 缓存查询
- **问题描述**：
  - `_performSnap()` 在每次 `mouseMoveEvent` 中调用 `_getGuideExtentMin/Max`
  - 这两个方法每次都**遍历整个场景的所有 item**（`self.scene().items()`）
  - 一次鼠标移动最多触发 4 次全量遍历（X/Y 各 min/max 一次），当画布上有几十张图片时开销巨大
  - 对比之下，`_buildSnapGuides` + `_findNearestGuide` 使用缓存 + 二分查找，已经是 O(log n)
- **修复方案**：
  - 在 `_buildSnapGuides()` 中新增 `_snap_item_rects` 字典，缓存每个参考物体的 `sceneBoundingRect()`
    ```python
    self._snap_item_rects = {}  # {item_id: QRectF}
    self._snap_item_rects[iid] = QRectF(r)  # 缓存边界矩形
    ```
  - `_getGuideExtentMin/Max` 不再遍历 `scene().items()`，改为遍历已排序的 guides 列表 + 查询缓存 rects
    ```python
    cached_rects = getattr(self, '_snap_item_rects', {})
    guides = self._snap_x_guides if axis == 'y' else self._snap_y_guides
    for guide_val, iid in guides:
        if abs(guide_val - snap_value) < 1:

  - 这符合 PureRef 等专业参考图工具的交互惯例

  ```

  |----------|---------------|------|
  | 组 | 解散组 | 组框消失，组内图片保留在画布上 |
  | 图片 | 删除图片 | 图片从画布上移除 |
  | 组 + 图片 | 解散组 + 删除图片 | 选中的组解散，选中的图片删除 |

    - 移除成员图片的 `group_id` 属性
    - 从场景和 `self.groups` 字典中移除组对象
    - 组成员图片保留在画布上，不再随组移动
    - 支持撤销（`UngroupCommand`）
  - **图片处理**：对选中的图片调用 `DeleteItemsCommand` 删除
    - 支持撤销，可通过 `Ctrl+Z` 恢复
- **交互细节**：
  - 解散组后，原组成员图片自动脱离组关系，成为独立图片
  - 解散操作支持撤销，撤销后组和成员关系恢复
  - 删除图片操作同样支持撤销

## 2026-02-25 周二 - 弹簧物理动画模型（Windows 端）

### 上午 10:39 - 重构：智能整理动画从 ease-out 改为阻尼弹簧模型
- **变更动机**：
  - macOS 端使用 SwiftUI `.spring(response: 0.5, dampingFraction: 0.75)` 动画有自然的 overshoot 回弹效果，"Q弹灵动"
  - Windows 端此前使用 `ease_out_cubic` 缓动函数，动画效果"平铺直叙"，缺乏生命力
  - 双端动画体验不一致，需统一
- **新方案**：实现与 macOS SwiftUI `.spring()` 等效的阻尼弹簧缓动函数
- **数学模型**：
  - **阻尼比** ζ = 0.75（欠阻尼，产生 overshoot 回弹）
  - **固有频率** ω₀ = 2π / response = 2π / 0.5 ≈ 12.57 rad/s
  - **阻尼振荡频率** ωd = ω₀ × √(1 - ζ²) ≈ 8.31 rad/s
  - **位移公式**：`x(t) = 1 - e^(-ζω₀t) × (cos(ωd·t) + (ζω₀/ωd) × sin(ωd·t))`
  - **物理意义**：指数衰减包络下的阻尼振荡，欠阻尼时会有轻微"冲过头"再回弹的效果
- **动画参数**：
  | 参数 | 旧值 | 新值 | 说明 |
  |------|------|------|------|
  | `anim_frames` | 20 | 45 | 动画总帧数 |
  | 总时长 | ~320ms | ~720ms | 45帧 × 16ms，与 macOS spring(response:0.5) 视觉接近 |
  | 缓动函数 | `ease_out_cubic` | `spring_damped` | 三次方缓出 → 阻尼弹簧 |
- **代码变更**：
  ```python
  def spring_damped(t, damping_ratio=0.75, frequency=2.0 * math.pi / 0.5):
      if t >= 1.0:
          return 1.0
      zeta = damping_ratio
      omega0 = frequency
      omega_d = omega0 * math.sqrt(1.0 - zeta * zeta)
      decay = math.exp(-zeta * omega0 * t)
      cos_part = math.cos(omega_d * t)
      sin_part = (zeta * omega0 / omega_d) * math.sin(omega_d * t)
      return 1.0 - decay * (cos_part + sin_part)
  ```
- **影响文件**：

  - **斥力（Repulsion）**：检测两两 AABB 重叠，沿最小重叠轴推开，消除重叠
  - **向心引力（Attraction）**：所有图片向共同质心聚拢，保持布局紧凑
  - **阻尼（Damping）**：每帧速度衰减 ×0.85，确保模拟稳定收敛
- **关键参数**（双端一致）：
  | 参数 | 值 | 说明 |
  |------|---|------|
  | `spacing` | 12px/pt | 图片间最小期望间距 |
  | `repulsion_strength` | 0.8 | 斥力系数（建议 0.3~1.5） |
  | `attraction_strength` | 0.02 | 引力系数（建议 0.005~0.1） |
  | `damping` | 0.85 | 速度阻尼（0~1） |
  | `max_iterations` | 80 | 最大迭代次数 |
  | `convergence_threshold` | 0.5 | 收敛阈值 |
  | `max_force` | 200.0 | 单步最大力（安全阀） |
- **打组兼容**：整理完成后自动更新受影响组的边界（`update_group_bounds` / `updateGroupBounds`），避免图片溢出组框
- **偏移校正**：整理后布局质心自动对齐到原始选区质心，图片不会跑偏
- **依赖优化**：Windows 端移除 `rectpack` 第三方依赖，算法完全自包含
- **影响文件**：
  - Windows: `Views/MainWindow.py`（`organize_items` 方法重写）、`requirements.txt`（移除 rectpack）

## 2026-02-09 周一 - 双击编辑组名称：彻底修复（双端）

### 下午 17:24 - 修复：macOS 端双击组名称无法编辑
- **问题**：macOS 端（SwiftUI）双击组名称标签无法触发编辑
- **根本原因**：名称标签使用 `.offset(y: -group.fontSize - 12)` 定位到组框上方。SwiftUI 中 `.offset()` **仅改变视觉位置，不改变 hit testing 区域**。用户双击看到的标签位置时，点击事件实际命中的是原始位置（组框内部），而非标签的视觉位置。同时 `GroupView` 上的 `DragGesture` 和 `onTapGesture` 也会抢占名称标签上的双击手势
- **修复方案**：
  - 将名称标签从 ZStack 内部移出，改用 `.overlay(alignment: .topLeading)` 方式定位
  - 使用 `.alignmentGuide(.top) { d in d[.bottom] + 4 }` 将标签正确推到组框上方，且 hit testing 区域与视觉位置一致
  - 名称标签上独立绑定 `onTapGesture(count: 2)` 和 `onTapGesture(count: 1)`，不受 GroupView 上的 DragGesture 干扰
- **影响文件**：
  - macOS: `Views/CanvasView.swift`（`GroupView` 结构体重构名称标签定位方式）

### 下午 17:19 - 修复：Win11 端双击组名称无法编辑
- **问题**：Win11 端（Qt）双击组名称标签始终无法触发编辑
- **根本原因**：`GroupItem` 的 `zValue` 为 `-100`（在图片下方），且 `RefView` 设置了 `RubberBandDrag` 模式。在此模式下，`QGraphicsView` 会优先将空白区域的鼠标事件用于橡皮筋框选，根本不会将事件分发给底层的 `GroupItem`

### 下午 17:48 - 修复 & 优化：组名称交互与布局
- **修复问题（Win11）**：双击组名称无法快速编辑（彻底修复）
