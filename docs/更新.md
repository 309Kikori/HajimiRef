## 2026-02-25 周二 - 弹簧物理动画模型（Windows 端）

### 上午 10:39 - 重构：智能整理动画从 ease-out 改为阻尼弹簧模型
- **变更动机**：
  - macOS 端使用 SwiftUI `.spring(response: 0.5, dampingFraction: 0.75)` 动画有自然的 overshoot 回弹效果，"Q弹灵动"
  - Windows 端此前使用 `ease_out_cubic` 缓动函数，动画效果"平铺直叙"，缺乏生命力
  - 双端动画体验不一致，需统一
- **新方案**：实现与 macOS SwiftUI `.spring()` 等效的阻尼弹簧缓动函数
- **数学模型**：
  - **阻尼比** ζ = 0.75（欠阻尼，产生 overshoot 回弹）
  - **固有频率** ω₀ = 2π / response = 2π / 0.5 ≈ 12.57 rad/s
  - **阻尼振荡频率** ωd = ω₀ × √(1 - ζ²) ≈ 8.31 rad/s
  - **位移公式**：`x(t) = 1 - e^(-ζω₀t) × (cos(ωd·t) + (ζω₀/ωd) × sin(ωd·t))`
  - **物理意义**：指数衰减包络下的阻尼振荡，欠阻尼时会有轻微"冲过头"再回弹的效果
- **动画参数**：
  | 参数 | 旧值 | 新值 | 说明 |
  |------|------|------|------|
  | `anim_frames` | 20 | 45 | 动画总帧数 |
  | 总时长 | ~320ms | ~720ms | 45帧 × 16ms，与 macOS spring(response:0.5) 视觉接近 |
  | 缓动函数 | `ease_out_cubic` | `spring_damped` | 三次方缓出 → 阻尼弹簧 |
- **代码变更**：
  ```python
  def spring_damped(t, damping_ratio=0.75, frequency=2.0 * math.pi / 0.5):
      if t >= 1.0:
          return 1.0
      zeta = damping_ratio
      omega0 = frequency
      omega_d = omega0 * math.sqrt(1.0 - zeta * zeta)
      decay = math.exp(-zeta * omega0 * t)
      cos_part = math.cos(omega_d * t)
      sin_part = (zeta * omega0 / omega_d) * math.sin(omega_d * t)
      return 1.0 - decay * (cos_part + sin_part)
  ```
- **影响文件**：
  - Windows: `Views/MainWindow.py`（`organize_items` 方法：删除 `ease_out_cubic`，新增 `spring_damped` 函数，调整 `anim_frames` 为 45）

### 下午 14:28 - 优化：智能整理时组边界与图片同步过渡动画
- **问题描述**：智能整理涉及组的情况下，图片有流畅的 ease-out 缓动动画，但组框只在动画结束后瞬间改变大小，视觉上不美观不优雅
- **根本原因**：
  - **Win11 端**：`animate_frame()` 每帧只插值图片的 `setPos()`，动画结束后（`t >= 1.0`）才一次性调用 `update_group_bounds()` 瞬间更新组大小

- **修复方案**：
  - **Win11 端**：在动画开始前预计算每个受影响组的起始 rect 和目标 rect（基于成员图片的目标位置），然后在 `animate_frame()` 的每一帧中同时插值图片位置和组的 rect（使用相同的 ease-out-cubic 缓动函数），实现完全同步的过渡动画
  - **macOS 端**：将组边界更新（`updateGroupBounds`）移入与图片位置更新相同的 `withAnimation(.spring())` 块中，SwiftUI 自动为组的 x/y/width/height 属性变化生成同步的弹性过渡动画

## 2026-02-13 周五 - 打组功能架构优化：几何交集实时判定（Position is Truth）

### 下午 13:18 - 优化：组成员判定从"显式列表维护"改为"实时几何交集计算"
- **改造动机**：
  - 此前使用 `member_ids`（`id(item)` 内存地址列表）显式维护成员关系，需要在每一个操作入口（拖拽、加载、撤销、打组、解组、智能整理...）中同步更新，任何一个入口遗漏就导致状态不一致
  - `id(item)` 不可序列化 → 加载看板后 `member_ids` 为空 → 组成员关系丢失
  - 已尝试多次补丁修复，但双重状态（`item.group_id` + `member_ids`）的根本矛盾无法消除
- **新方案：几何交集实时判定（Position is Truth）**：
  - 完全删除 `member_ids` 列表，不再维护任何显式成员列表
  - 新增 `GroupItem.get_members_by_intersection(threshold=0.05)` 方法：遍历场景中所有 `RefItem`，计算每张图片与组框的交集面积。**交集面积 ≥ 图片面积 × 5% 即为组成员**
  - 拖拽开始时（`mousePressEvent`）调用一次交集判定并缓存成员列表 `_drag_members`，拖拽过程中直接使用缓存（无需每帧重新计算）
  - 所有组操作（`update_group_bounds`、`check_images_in_group_bounds`、`check_image_out_of_group`、`ungroup`、`record_group_move_action` 等）统一通过 `_get_group_members()` → `get_members_by_intersection()` 获取成员
- **核心优势**：
  - **无状态同步问题**：不存在双重状态，位置即真相
  - **无序列化问题**：加载看板后自动通过几何位置恢复成员关系
  - **无撤销/重做状态丢失问题**：撤销/重做后位置改变，成员关系自然随之变化
  - **代码量减少**：删除 `add_member`/`remove_member` 中的列表维护逻辑
- **5% 阈值说明**：
  - 图片仅有极小角搭在组框上时不算成员，避免误判
  - 图片有 5% 以上面积与组框重叠时算成员，比"中心点判定"更直观
- **影响文件**：
  - `Views/Canvas.py`（`GroupItem`：新增 `get_members_by_intersection()`，改造 `add_member`/`remove_member`/`update_bounds`/`mousePressEvent`/`mouseMoveEvent`/`mouseReleaseEvent`）
  - `Views/MainWindow.py`（改造 `_get_group_members()`、`check_images_in_group_bounds()`、`check_image_out_of_group()`、`group_selected_items()`）

## 2026-02-13 周五 - 打组功能 BUG 修复：成员判定错误 + 拖拽延迟

### 上午 09:34 - 修复：组拖拽时非成员图片跟随移动 + 移动延迟感
- **问题描述**：
  1. 有些时候图片并不在组中（已被移出或从未加入），但移动组时图片依旧跟随移动
  2. 移动组时成员图片有明显的延迟感，不跟手
- **根本原因**：
  1. **成员判定错误（Win11 端）**：组拖拽时通过遍历 `scene.items()` + `item.group_id == self.group_id` 来查找成员。存在"双重状态"——`item.group_id`（图片自身记录所属组）和 `GroupItem.member_ids`（组记录所属图片）两套数据不一致时，会导致已从组移除的图片仍被当作成员移动
  2. **拖拽延迟一帧（Win11 端）**：`mouseMoveEvent` 中先计算 `delta = self.pos() - self._undo_start_pos`，再调用 `super().mouseMoveEvent(event)`。但 Qt 的 `super()` 才会更新 `pos()`，所以 delta 总是基于**上一帧**的位置——成员图片永远落后组框一帧
  3. **全场景遍历性能问题**：每帧拖拽都遍历 `scene.items()` 查找成员，当场景中图片数量多时造成性能开销
- **修复方案（Win11 端）**：
  - **统一成员权威来源为 `member_ids`**：新增 `_get_group_members()` 辅助方法，所有需要查找组成员的地方（`update_group_bounds`、`ungroup`、`record_group_move_action`、`check_images_in_group_bounds` 等）统一通过 `member_ids` 查找，不再依赖 `item.group_id`
  - **消除拖拽延迟**：将 `super().mouseMoveEvent(event)` 调到成员位移逻辑**之前**调用，这样 `self.pos()` 在计算 delta 时已是最新位置，成员图片与组框完全同步
  - **缓存成员引用**：拖拽开始时（`mousePressEvent`）构建 `_members_refs` 字典缓存成员 item 的引用，拖拽过程中直接遍历缓存而非遍历全场景
- **macOS 端状态**：macOS 端组拖拽已正确使用 `group.memberIds`，且 SwiftUI 声明式 UI 不存在帧延迟问题，无需修改
- **影响文件**：
  - `Views/Canvas.py`（GroupItem 的 `mousePressEvent`、`mouseMoveEvent`、`mouseReleaseEvent`）
  - `Views/MainWindow.py`（新增 `_get_group_members` 方法，重构所有组成员查找逻辑）

## 2026-02-11 周三 - 智能整理升级：物理模拟布局算法

### 下午 17:55 - 修复：物理模拟布局残留重叠问题
- **问题描述**：智能整理后图片之间仍有重叠
- **根因分析**（三个缺陷同时导致）：
  1. **向心引力与斥力博弈**：向心引力在每一帧都把图片往质心拉，而斥力仅在重叠时起作用。最后几帧图片被拉回重叠，但速度已低于收敛阈值，模拟就停止了——重叠未完全消除就收敛
  2. **阻尼过强 + 迭代不足**：damping=0.85 导致斥力产生的推开速度在一帧内就被吃掉大半，80 次迭代不够收敛
  3. **缺少最终去重叠保障**：模拟结束后没有最终碰撞检测，残留重叠无法修正
- **修复方案**（双端一致）：

## 2026-02-09 周一 - 双击编辑组名称：彻底修复（双端）

### 下午 17:24 - 修复：macOS 端双击组名称无法编辑
- **问题**：macOS 端（SwiftUI）双击组名称标签无法触发编辑
- **根本原因**：名称标签使用 `.offset(y: -group.fontSize - 12)` 定位到组框上方。SwiftUI 中 `.offset()` **仅改变视觉位置，不改变 hit testing 区域**。用户双击看到的标签位置时，点击事件实际命中的是原始位置（组框内部），而非标签的视觉位置。同时 `GroupView` 上的 `DragGesture` 和 `onTapGesture` 也会抢占名称标签上的双击手势
- **修复方案**：
  - 将名称标签从 ZStack 内部移出，改用 `.overlay(alignment: .topLeading)` 方式定位
  - 使用 `.alignmentGuide(.top) { d in d[.bottom] + 4 }` 将标签正确推到组框上方，且 hit testing 区域与视觉位置一致
  - 名称标签上独立绑定 `onTapGesture(count: 2)` 和 `onTapGesture(count: 1)`，不受 GroupView 上的 DragGesture 干扰
- **影响文件**：
  - macOS: `Views/CanvasView.swift`（`GroupView` 结构体重构名称标签定位方式）

### 下午 17:19 - 修复：Win11 端双击组名称无法编辑
- **问题**：Win11 端（Qt）双击组名称标签始终无法触发编辑

## A1.0.2 (2025-11-28) - 性能与体验大修
### 新增功能
*   **始终置顶**: 在设置菜单中增加了"始终置顶 (Always on Top)"选项，方便在绘画时参考。
*   **增强粘贴**: 优化了 `Ctrl+V` 逻辑，现在不仅支持粘贴截图，还支持从资源管理器直接复制文件后粘贴。

### 优化
*   **视口裁剪 (Viewport Clipping)**: 重写了图像渲染管线。现在只渲染屏幕可见区域的图像切片。
    *   *效果*: 彻底解决了图片放大倍数过大时（如 >500%）导致的严重卡顿和内存溢出问题。
*   **动态重采样 (Dynamic Resampling)**: 
    *   滚动/拖拽时使用 `Nearest Neighbor` 插值，保证 60fps 流畅度。
    *   停止操作 200ms 后自动切换为 `Lanczos/Bilinear` 插值，保证画质清晰。
*   **汉化补全**: 修复了界面中遗漏的英文文本（如 Exit, Always on Top）。

## A1.0.1 (2025-11-28) - 基础功能完善
### 新增功能
*   **多语言支持**: 实现了简易的 i18n 框架，支持英文和简体中文切换。
*   **保存/读取**: 支持将当前看板保存为 `.sref` (JSON) 文件，图片数据被 Base64 编码内嵌其中，方便单文件传输。

### 修复
*   修复了初始版本中图片缩放比例计算错误的问题。
